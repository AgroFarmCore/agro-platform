# ADR-001: Monorepo + Modular Monolith

- Status: Accepted
- Date: 2026-04-01

## Context

Новый продукт в агро-сфере должен включать:

- публичный сайт;
- кабинет фермера;
- кабинет поставщика;
- backoffice;
- рекомендации;
- базовый учёт и аналитику;
- фоновые задачи;
- API и интеграции.

Команда небольшая: 5 разработчиков и tech lead, который тоже пишет код. На старте критичны:

- быстрый onboarding;
- понятные boundaries;
- простой code review;
- минимальная организационная стоимость изменений;
- отсутствие ранней микросервисной сложности.

## Decision

Выбираем:

- один `pnpm` monorepo;
- `Turborepo` для task orchestration и caching;
- несколько runtime apps: `web`, `portal`, `backoffice`, `worker`;
- modular monolith как способ организации предметной логики;
- доменная и application логика концентрируется в `packages/modules`;
- Prisma schema и DB tooling живут в `packages/database`;
- очереди и job contracts живут в `packages/jobs`.

## Rationale

Это решение лучше всего подходит текущему масштабу команды и продукта:

- не создаёт сетевую, деплойную и контрактную сложность микросервисов;
- даёт хорошие границы через packages и import rules;
- позволяет переиспользовать типы, UI, policies, contracts и test helpers;
- упрощает AI-assisted coding: Codex легче ориентируется в явных package boundaries и публичных API;
- даёт возможность позже выделить отдельные сервисы только для реально перегруженных участков.

## Consequences

Положительные:

- высокая скорость разработки;
- меньше интеграционного шума;
- проще поддерживать единые стандарты;
- меньше дублирования между кабинетами, worker и backoffice.

Негативные и компромиссы:

- нужен контроль import boundaries, иначе monorepo быстро расползётся;
- worker и apps должны строго переиспользовать модульную логику, иначе появится скрытое дублирование;
- при сильном росте нагрузки часть модулей может потребовать вынесения в отдельные сервисы позже.

## What We Explicitly Reject

- микросервисную архитектуру на старте;
- несколько репозиториев для web, admin, worker и shared packages;
- раннее появление `shared/utils` как общего склада кода;
- прямой Prisma-доступ из любого места без module repositories;
- отдельный CMS runtime без подтверждённой editorial потребности.

## Review Trigger

Решение пересматривается, если одновременно выполняются хотя бы два условия:

- команда выросла заметно больше текущего состава;
- один из runtime apps требует независимого релизного цикла каждую неделю и отдельного ownership;
- внешние integrations/API требуют отдельных SLA и сетевой изоляции;
- worker/recommendation subsystem упирается в масштабирование и жизненный цикл, сильно отличный от основной платформы.
